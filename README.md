# Job Launcher

**Job Launcher 是一个面向求职场景的桌面自动化工具。**

它的定位不是替用户做判断，而是把求职平台上的重复操作沉淀成一套可以配置、执行和复盘的流程：职位筛选、自动开聊、已读不回提醒、职位记录、公司记录和本地数据沉淀。

在求职密集期，真正消耗精力的往往不是单次投递，而是每天反复打开平台、刷新职位、判断岗位是否合适、点开详情、发起沟通、记录进展、等待回复。Job Launcher 尝试把这些高重复、低判断密度的动作自动化，让用户把精力更多放在简历优化、岗位判断和面试准备上。

---

## 适合解决什么问题

| 场景 | 解决的问题 |
|---|---|
| 求职冷启动 | 批量浏览职位，快速获得市场反馈 |
| 岗位筛选 | 按城市、职位、薪资、经验、关键词等条件过滤职位 |
| 自动开聊 | 对符合条件的岗位自动发起沟通 |
| 已读不回 | 对已读未回复的会话进行提醒或二次跟进 |
| 职位沉淀 | 记录已沟通、已跳过、不合适职位和公司 |
| 求职复盘 | 通过本地数据观察岗位质量、沟通进展和回复情况 |

## 核心功能

### 1. 自动开聊

Job Launcher 可以按照配置好的职位来源和筛选条件，自动浏览职位列表，并对符合条件的岗位发起沟通。

<img width="1309" alt="Job Launcher 自动开聊界面" src="https://github.com/user-attachments/assets/fda33e53-347b-453d-a791-5a1dcd506605" />

常见筛选维度包括：

- 工作城市；
- 薪资范围；
- 工作经验；
- 职位名称；
- 职位类型；
- 职位描述关键词；
- 招聘方活跃度；
- 公司黑名单或不合适规则。

不符合条件的岗位可以按策略处理：

| 处理方式 | 适用情况 |
|---|---|
| 平台内标记不合适 | 明确不想再次看到该职位 |
| 本地数据库标记不合适 | 不想影响平台推荐，但希望后续自动跳过 |
| 本次运行跳过 | 只是当前任务不处理，下次仍可重新判断 |

实际运行时，可以先选择职位来源和筛选条件，再由工具自动进入职位列表、查看职位详情，并按规则决定是否发起沟通。

<img width="1680" height="1050" alt="Job Launcher 职位来源与筛选配置" src="https://github.com/user-attachments/assets/cbdea071-ec28-49b0-af12-1018aa9579a9" />

### 2. 已读不回提醒

Job Launcher 可以扫描沟通列表，对已读未回复的会话进行跟进。

<img width="1024" alt="Job Launcher 已读不回提醒配置" src="https://github.com/user-attachments/assets/51a43b0f-118d-4b0a-957c-40a5367e29e4" />

支持的控制项包括：

- 跟进时限；
- 跟进间隔；
- 单个会话重复提醒控制；
- 固定提醒内容；
- 基于大语言模型生成提醒内容。

这部分适合用在求职沟通中后段，帮助用户减少遗漏，同时避免过度频繁地重复打扰。

### 3. 登录助手

工具提供登录辅助流程，降低手工处理 Cookie、登录凭证和浏览器配置的操作成本。

<img width="1680" alt="Job Launcher 登录助手" src="https://github.com/user-attachments/assets/b5d02385-f1c0-4045-b01a-66c5f747fec5" />

登录凭据会保存在本地，用于后续自动化访问。使用前需要确认当前设备安全，不要在公共电脑或不可信环境中保存账号凭据。

### 4. 大语言模型配置

Job Launcher 支持配置大语言模型，用于生成部分沟通跟进内容。

<img width="576" alt="Job Launcher 大语言模型配置" src="https://github.com/user-attachments/assets/ef91642f-b0b0-438f-8c66-e650f0de139b" />

适合的使用方式是：

- 根据当前职位和沟通上下文生成提醒；
- 让跟进内容更自然；
- 避免每次都发送完全相同的话术；
- 在模型不可用时切换备用配置。

如果默认提示词不适合当前求职表达，也可以在工具里编辑已读不回提醒的提示词模板。

<img width="1139" alt="Job Launcher 已读不回提醒提示词编辑" src="https://github.com/user-attachments/assets/db956573-61c0-44d4-b588-5c04fe941f84" />

配置模型后，可以先用模拟功能测试生成效果，再决定是否用于实际跟进。

<img width="600" alt="Job Launcher 已读不回提醒模拟" src="https://github.com/user-attachments/assets/3b2776bd-ae5b-48d1-a01c-0819509d96a1" />

### 5. 本地记录与复盘

工具会在本地记录运行过程中产生的数据，包括职位、公司、沟通、标记不合适、自动开聊记录等。

这些数据的价值在于后续复盘：

- 哪些岗位类型回复率更高；
- 哪些关键词更容易命中合适职位；
- 哪些公司或岗位应该加入跳过规则；
- 自动开聊是否带来了有效沟通；
- 哪些跟进动作值得保留。

### 6. 配置模板

Job Launcher 也提供了一些配置模板，用来降低第一次配置的成本。比如期望公司、职位详情筛选、大语言模型等，都可以先基于模板调整，再进入实际运行。

## 使用前需要知道的风险

Job Launcher 是浏览器自动化辅助工具，使用前需要明确以下风险：

1. 求职平台可能不鼓励或限制自动化行为，账号可能受到风控、限制使用或强制退出登录。
2. 平台页面经常改版，自动化脚本可能因为页面结构变化而失效。
3. 工具需要在本地保存登录凭据，请不要把本地配置文件、Cookie、数据库文件发送给他人。
4. 自动开聊不等于有效求职，岗位质量、简历匹配度、沟通表达和面试准备仍然需要人工判断。
5. 建议控制运行频率和开聊数量，不要把工具用于骚扰、刷量或任何违反平台规则的行为。

我更推荐把它当成“求职执行助手”，而不是完全替代人工判断的工具。

## 本地运行

项目基于 Electron、Vue、Puppeteer 和 pnpm workspace。

```bash
pnpm install
pnpm --filter ./packages/ui dev
```

常用命令：

```bash
pnpm --filter ./packages/ui build
pnpm --filter ./packages/ui typecheck
pnpm --filter ./packages/ui lint
```

构建桌面安装包：

```bash
pnpm --filter ./packages/ui build:win
pnpm --filter ./packages/ui build:mac
pnpm --filter ./packages/ui build:linux
```

## 数据存储

默认本地运行数据会写入应用本地目录，常见数据包括：

| 类型 | 说明 |
|---|---|
| config | 运行配置、职位筛选配置、大模型配置 |
| storage | Cookie、localStorage、本地数据库 |
| cache | 浏览器下载、运行缓存 |

上传 Issue 或公开截图前，需要先检查是否包含 Cookie、简历、聊天记录、公司名单、个人身份信息等敏感内容。

## 项目定位

Job Launcher 更像一个求职自动化实验项目，重点在于把求职过程中的重复动作产品化：

```text
职位来源 -> 筛选规则 -> 自动浏览 -> 自动开聊 -> 跟进提醒 -> 本地记录 -> 求职复盘
```

这套链路背后的思路，也可以迁移到其他运营场景：

- 批量任务执行；
- 条件筛选；
- 自动化触达；
- 本地数据沉淀；
- 过程复盘；
- 规则持续优化。
